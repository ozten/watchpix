# watchpix

**A live-reloading image gallery server for remote and headless machines.**

`watchpix` watches a directory tree for image files, serves a web-based gallery sorted by recency, and live-updates the browser via WebSocket whenever images are created, modified, or deleted. It is designed for developers and AI/ML practitioners who work on headless VMs over SSH and need a frictionless way to see generated images without `scp`, X11 forwarding, or heavyweight remote desktop setups.

## Usage

```bash
# Watch current directory, serve on port 8080
watchpix

# Watch a specific directory on a custom port
watchpix ./output --port 3000

# Add extra directories to the deny list
watchpix ./workspace --deny vendor,assets/raw

# Then on your local machine, tunnel in:
ssh -L 8080:localhost:8080 user@remote-host

# Open http://localhost:8080 in your browser
```

## Typical workflow

1. You SSH into a remote VM (possibly through multiple hops via `ProxyJump`).
2. Agents, scripts, or training jobs are generating images on that machine.
3. You run `watchpix` in the project root (or any parent directory).
4. You SSH-tunnel the port to your local machine.
5. You open `localhost:8080` in a browser tab and leave it open.
6. Images appear in the gallery in real time as they are created or modified, newest first.
7. You click any image to view it full-resolution.

No `scp`. No `ls -altr ~/Downloads`. No context switching.

## Technical specification

### 1. Directory scanning

#### 1.1 Initial scan

On startup, `watchpix` performs a recursive walk of the root directory. For every file encountered:

1. Check if the file's parent directory (at any level) matches the deny list. If so, skip the entire subtree.
2. Check if the file extension matches the supported image extensions. If so, read the file's `mtime` and add it to the in-memory image index.
3. After the full walk completes, sort the index by `mtime` descending (newest first).

The initial scan and the "new directory discovered" scan (see §2.3) **must** share the same code path: a single `scan_directory(path, deny_list) -> Vec<ImageEntry>` function. This keeps filtering logic in exactly one place.

#### 1.2 Supported image extensions

```
png, jpg, jpeg, gif, webp, svg, bmp, tiff, tif, ico, avif
```

Extension matching is case-insensitive. SVG is included because browsers render it natively in `<img>` tags.

#### 1.3 Default deny list

The following directory **names** are denied by default. Matching is on the directory name component only, not the full path — so `node_modules` is denied wherever it appears in the tree.

```
node_modules
.git
.hg
.svn
.cache
.venv
.env
__pycache__
target
dist
build
vendor
.next
.nuxt
.output
.turbo
.parcel-cache
coverage
tmp
temp
```

Users can extend this list via the `--deny` CLI flag (comma-separated). User-specified entries are additive to the defaults. There is no mechanism to remove a default deny entry in v1.

#### 1.4 Image index data structure

The index is a `Vec<ImageEntry>` protected by a `tokio::sync::RwLock`, sorted by `mtime` descending.

```rust
struct ImageEntry {
    path: PathBuf,       // relative to the root directory
    mtime: SystemTime,   // last modification time
    size: u64,           // file size in bytes
}
```

At the scale this tool operates (hundreds to low thousands of images in a project tree), a `Vec` with linear scan for updates is sufficient. There is no need for a `HashMap` or `BTreeMap` for v1.

Operations on the index:

- **Add**: Push entry, re-sort or insert at correct position.
- **Update (modify)**: Find by path, update `mtime` and `size`, move to front.
- **Remove**: Find by path, remove entry.
- **Move to front**: Find by path, remove from current position, insert at index 0.

### 2. File watching

#### 2.1 Watcher setup

`watchpix` uses the `notify` crate with `RecommendedWatcher`. On startup, the watcher is registered on the root directory in `RecursiveMode::Recursive`.

On Linux, `inotify` (used by `RecommendedWatcher`) automatically watches new subdirectories created under a recursively-watched parent. On macOS, `FSEvents` also handles this natively. However, `watchpix` does **not** rely on this platform-specific behavior. It explicitly handles new directory events (see §2.3) to ensure correctness across all platforms and to handle the "scan existing contents" case that filesystem events alone will not cover.

#### 2.2 Event handling

The watcher handler dispatches on event kind and path type:

| Event | Path type | Action |
|---|---|---|
| Create | Image file | Add to index, broadcast `add` |
| Create | Directory | Filter check → watch → scan → broadcast (see §2.3) |
| Modify | Image file | Update mtime, move to front, broadcast `update` |
| Remove | Image file | Remove from index, broadcast `remove` |
| Remove | Directory | Remove all images under that path from index, broadcast `remove` for each |
| Rename | — | Treated as remove of old path + create of new path |

Events for non-image files and denied directories are silently discarded.

#### 2.3 New directory handling

When a `Create` event fires for a directory:

1. Check the directory name against the deny list. If denied, ignore entirely — do not recurse into it.
2. If allowed, register the watcher on the new directory **first**.
3. Then recursively scan the directory contents using the shared `scan_directory` function.
4. Add any discovered images to the index.
5. Broadcast additions to connected clients.

**Ordering rationale (race condition mitigation):** The watcher must be registered *before* the scan begins. If an image file lands in the new directory between the "directory created" event and the moment we start watching, we would miss it if we scanned first and watched second. By watching first and scanning second:

- Files that exist before watching starts are picked up by the scan.
- Files that arrive after watching starts are picked up by the watcher.
- Files that arrive during the gap between watch registration and scan completion may be detected by both the watcher and the scan. This produces duplicates, which are handled by deduplication (see §2.4).

#### 2.4 Deduplication

Before adding an image to the index or broadcasting an event, check whether the path already exists in the index:

- If the path exists with the **same** `mtime`: no-op. This is a duplicate event from the race condition described in §2.3, or a spurious watcher event. Discard silently.
- If the path exists with a **different** `mtime`: treat as a modification. Update the entry and broadcast an `update` event.
- If the path does not exist: add it and broadcast an `add` event.

#### 2.5 Debouncing

Many editors and tools write files in multiple stages (create temp file, write contents, rename into place) or modify files rapidly in sequence. `watchpix` applies a debounce window of **100ms** per path: if multiple events fire for the same path within 100ms, only the last event is processed.

This is implemented via a `HashMap<PathBuf, tokio::time::Instant>` that tracks the last event time per path. When an event arrives, it is buffered; after the debounce window expires with no further events for that path, the event is processed.

### 3. Web server

#### 3.1 HTTP routes

| Route | Method | Description |
|---|---|---|
| `GET /` | GET | Serves the gallery HTML page with inline CSS and JS |
| `GET /api/images` | GET | Returns the full image index as JSON |
| `GET /image/*path` | GET | Serves the raw image file from disk with appropriate `Content-Type` |
| `GET /ws` | GET | WebSocket upgrade endpoint |

#### 3.2 Gallery page (`GET /`)

A single self-contained HTML page with inline CSS and JavaScript. No external dependencies. No framework.

The page:

1. On load, fetches `/api/images` to get the current index.
2. Renders a responsive grid of image cards, each showing a thumbnail-sized preview, the relative file path, the timestamp, and the file size.
3. Opens a WebSocket connection to `/ws`.
4. On receiving WebSocket messages, manipulates the DOM directly to add, move, or remove image cards with a brief CSS transition so the user can see what changed.
5. Supports clicking an image card to open the full-resolution image in a lightbox overlay.
6. If the WebSocket connection drops, attempts to reconnect with exponential backoff (1s, 2s, 4s, ... capped at 30s). Displays a subtle "reconnecting..." indicator in the UI.

The page should include a connection status indicator (green dot = connected, yellow = reconnecting, red = disconnected).

#### 3.3 Images API (`GET /api/images`)

Returns JSON:

```json
{
  "root": "/home/user/project",
  "count": 42,
  "images": [
    {
      "path": "output/run-42/loss_curve.png",
      "url": "/image/output/run-42/loss_curve.png",
      "mtime": 1709500000,
      "size": 48230
    }
  ]
}
```

Images are sorted by `mtime` descending (newest first). No pagination in v1 — at the expected scale (hundreds of images), returning the full list is fine.

#### 3.4 Image serving (`GET /image/*path`)

Serves the raw image file from disk. The path is resolved relative to the root directory. Path traversal attacks (`../`) must be rejected — resolve the full path and verify it is a descendant of the root directory before serving.

Content-Type is set based on the file extension:

| Extension | Content-Type |
|---|---|
| png | image/png |
| jpg, jpeg | image/jpeg |
| gif | image/gif |
| webp | image/webp |
| svg | image/svg+xml |
| bmp | image/bmp |
| tiff, tif | image/tiff |
| ico | image/x-icon |
| avif | image/avif |

Cache-Control is set to `no-cache` — the browser should revalidate on every request. This ensures modified images are always fresh.

#### 3.5 WebSocket protocol (`GET /ws`)

After the WebSocket handshake, the server sends JSON messages to the client. The client does not send messages to the server (unidirectional).

**Message types:**

```json
{"type": "add", "image": {"path": "...", "url": "...", "mtime": ..., "size": ...}}
```

```json
{"type": "update", "image": {"path": "...", "url": "...", "mtime": ..., "size": ...}}
```

```json
{"type": "remove", "path": "..."}
```

On `add`: the client prepends a new image card at the top of the gallery.

On `update`: the client finds the existing card by path, updates its thumbnail (cache-bust by appending `?t=<mtime>` to the image URL), and moves it to the top of the gallery.

On `remove`: the client finds the card by path and removes it from the DOM.

#### 3.6 WebSocket client management

The server maintains a set of active WebSocket connections. When a filesystem event produces a broadcast, the message is sent to all connected clients. Dead connections are cleaned up on send failure.

This is implemented via a `tokio::sync::broadcast` channel. The watcher task sends events to the broadcast channel. Each WebSocket handler task subscribes to the broadcast channel and forwards messages to its client.

### 4. CLI interface

```
watchpix [OPTIONS] [ROOT]

Arguments:
  [ROOT]    Directory to watch [default: .]

Options:
  -p, --port <PORT>     Port to listen on [default: 8080]
  -b, --bind <ADDR>     Address to bind to [default: 127.0.0.1]
  -d, --deny <DIRS>     Additional directories to deny, comma-separated
  -v, --verbose         Enable verbose logging
  -h, --help            Print help
  -V, --version         Print version
```

**Default bind address is `127.0.0.1`**, not `0.0.0.0`. Since the intended use is over an SSH tunnel, binding to localhost only is the secure default. Users who need LAN access can explicitly pass `--bind 0.0.0.0`.

On startup, `watchpix` prints:

```
watchpix v0.1.0
Watching: /home/user/project
Found: 42 images
Denying: node_modules, .git, target, ... (18 patterns)
Server: http://127.0.0.1:8080

Tip: ssh -L 8080:localhost:8080 user@host
```

### 6. Not in scope for v1

The following are explicitly deferred:

- **Thumbnails / image resizing.** In v1, the browser loads full images and CSS handles display sizing. The `<img>` tags use `loading="lazy"` to avoid loading off-screen images. If performance becomes an issue with very large images, thumbnail generation via the `image` crate can be added in v2.
- **Authentication.** The tool binds to localhost by default and is accessed via SSH tunnel. No auth layer is needed.
- **HTTPS.** Same rationale — SSH provides the encrypted transport.
- **Persistent state.** The index is rebuilt from disk on every startup. There is no database, no cache file, no state to corrupt.
- **Configuration file.** CLI flags are sufficient for v1. A `.watchpixrc` or similar can be added later.
- **Filtering by directory in the UI.** The gallery shows all images flat. Directory-based browsing or filtering can be added in v2.
- **Upload / delete from the UI.** This is a read-only viewer.
- **Video support.** Images only.

### 8. Error handling

- If the root directory does not exist or is not readable, exit with a clear error message.
- If the port is already in use, exit with a clear error message suggesting an alternative port.
- If a watched directory becomes inaccessible (permissions change, unmounted), log a warning and continue operating on the remaining tree.
- If an image file cannot be read when serving (deleted between index check and file read), return HTTP 404.
- If the `notify` watcher hits the `inotify` watch limit, log an error with instructions to increase it (`echo 65536 > /proc/sys/fs/inotify/max_user_watches`) and continue with best-effort watching.

### 9. Performance expectations

- Initial scan of a project with 10,000 files (500 images) should complete in under 1 second.
- WebSocket broadcast latency from filesystem event to browser update should be under 200ms (excluding debounce window).
- Memory usage should stay under 50MB for typical projects. The index itself is negligible; the main cost is `notify` watch descriptors.
- The tool should handle at least 10 concurrent browser connections without degradation.
